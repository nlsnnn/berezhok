## 1. Обзор проекта

### 1.1 Описание

**Бережок** — геолокационная платформа для оперативного фудшеринга с системой смарт-уведомлений. Российский аналог Too Good To Go.

Платформа помогает заведениям (кафе, пекарни, рестораны, магазины) продавать нераспроданные продукты со скидкой в конце дня, а клиентам — экономить и бороться с пищевыми отходами.

### 1.2 Целевая аудитория

**Клиенты (B2C):**
- Студенты, молодые специалисты
- Экономные потребители
- Эко-активисты (борьба с food waste)

**Партнёры (B2B):**
- Пекарни, кофейни
- Рестораны, кафе
- Продуктовые магазины
- Отели (завтраки)
- Цветочные

### 1.3 Ключевая ценность

**Для клиентов:**
- Экономия 50-70% на качественной еде
- Доступ к премиальным заведениям по низким ценам
- Вклад в борьбу с пищевыми отходами

**Для партнёров:**
- Дополнительная выручка вместо выброса продуктов
- Привлечение новых клиентов
- Улучшение репутации (экологичность)

## 2. Функциональные требования

### 2.1 Клиентское приложение (Flutter, iOS/Android)

#### Регистрация и авторизация
- Регистрация по номеру телефона (без email и пароля)
- Вход по SMS коду (6 цифр, TTL 5 минут)
- Опциональное заполнение профиля (имя)

#### Поиск заведений
- **Карта:** отображение заведений с активными боксами на карте (Yandex Maps)
- **Список:** каталог заведений с фильтрами:
  - По категориям (пекарня, кафе, ресторан, магазин, отель)
  - По расстоянию (от ближайших)
  - По рейтингу
  - По цене
- **Геолокация:** показ расстояния до заведения в км/м

#### Сюрприз-боксы
- Просмотр активных боксов у заведения:
  - Название ("Утренний бокс", "Вечерний сюрприз")
  - Оригинальная цена (зачёркнута)
  - Цена со скидкой
  - Время получения (например, 18:00-19:00)
  - Описание (примерное содержимое)
  - Доступное количество
- Бронирование и оплата бокса

#### Оплата
- Интеграция с ЮKassa
- Оплата банковской картой
- После оплаты → генерация кода получения

#### Получение заказа
- Отображение деталей заказа:
  - QR-код для сканирования
  - Буквенно-цифровой код (AB12CD34)
  - Адрес и время получения
  - Навигация до заведения
- Клиент приходит в указанное время
- Показывает QR/код сотруднику
- Сотрудник сканирует/вводит код → заказ переходит в статус "Выдаётся"
- **Подтверждение получения:**
  - Через 15 минут после сканирования → автоматическое завершение
  - Кнопка "Не получил" → открытие спора (если в течение 15 минут)

#### Отзывы
- После завершения заказа → возможность оставить отзыв:
  - Рейтинг 1-5 звёзд
  - Текстовый комментарий (опционально)
- Один отзыв на один заказ

#### Push-уведомления (смарт-уведомления)
- **Новый бокс рядом:** если в радиусе 2 км появился новый бокс
- **Заказ оплачен:** подтверждение оплаты
- **Заказ подтверждён партнёром:** готов к получению
- **Напоминание:** за 1 час до окончания окна получения
- **Скидки от любимых заведений**

#### История заказов
- Список всех заказов (прошлых и активных)
- Статусы: оплачен, подтверждён, завершён, отменён

### 2.2 Веб-панель для партнёров

#### Регистрация партнёра
- Партнёр подаёт заявку через форму:
  - Контактное лицо (имя, телефон, email)
  - Название бизнеса
  - Тип заведения (категория)
  - Адрес
  - Описание
- Администратор рассматривает заявку вручную
- При одобрении:
  - Создаётся партнёр + заведение
  - Создаётся владелец (owner) с временным паролем
  - Отправляется email с доступами

#### Онбординг партнёра
- Первый вход → смена пароля (обязательно)
- Заполнение данных:
  - Юридическая информация (ИНН, ОГРН, адрес)
  - Реквизиты для выплат (банк, счёт, БИК)
  - Настройка заведения (часы работы, фото, описание)
- Создание первого сюрприз-бокса
- После заполнения → статус "Активен"

#### Управление сюрприз-боксами
- Создание бокса:
  - Название
  - Описание (что может попасть)
  - Оригинальная цена
  - Цена со скидкой
  - Время получения (начало-конец)
  - Количество доступных боксов
  - Фото (опционально)
- **Лимит:** 3-5 активных боксов одновременно
- Редактирование, деактивация боксов
- Быстрое изменение количества доступных боксов

#### Управление заказами

**Новые заказы (требуют подтверждения):**
- После оплаты клиентом → заказ появляется в разделе "Новые"
- **Дедлайн подтверждения:** 30 минут
- Партнёр должен подтвердить заказ (кнопка "Подтвердить")
- Если не подтвердил в течение 30 минут → автоматическая отмена + возврат денег клиенту

**Подтверждённые заказы:**
- Список заказов на сегодня/завтра
- Поиск заказа по коду
- Сканирование QR-кода клиента (если веб-камера доступна)
- Выдача заказа → кнопка "Выдано"

**История заказов:**
- Все прошлые заказы
- Фильтры по дате, статусу

#### Команда (сотрудники)
- Владелец может добавлять сотрудников:
  - Email, имя
  - Роль: manager (может управлять заказами) | employee (только выдача)
  - Привязка к конкретной точке (если несколько)
- Отправка приглашения с временным паролем на email

#### Финансы
- **Статистика:** выручка за день/неделю/месяц, количество заказов
- **Выписки:** еженедельные отчёты по выплатам
- **Текущий баланс:** сумма к выплате
- **Комиссия сервиса:** 
  - Стандартная: 20%
  - Промо-период (первые 3 месяца): 10%
  - VIP (>100 заказов/месяц): 15%

#### Отзывы
- Просмотр отзывов клиентов
- Средний рейтинг заведения
- Распределение оценок (1-5 звёзд)

### 2.3 Админ-панель

#### Управление партнёрами
- **Заявки на регистрацию:**
  - Просмотр заявок
  - Одобрение/отклонение с комментарием
  - При одобрении → автосоздание партнёра + владельца
- **Список партнёров:**
  - Просмотр всех партнёров
  - Фильтры по статусу (активен, заблокирован, ожидает документы)
  - Просмотр деталей партнёра (заведения, финансы, заказы)
  - Блокировка/разблокировка партнёра
- **Метрики:**
  - Процент пропущенных подтверждений заказов
  - Если >20% → предупреждение
  - Если >50% → временная блокировка

#### Управление пользователями
- Просмотр клиентов
- Поиск по телефону/имени
- Просмотр истории заказов клиента
- Блокировка пользователей (при жалобах)

#### Споры (Disputes)
- **Список споров:**
  - Открытые, на рассмотрении, решённые
- **Просмотр спора:**
  - Детали заказа
  - Комментарий клиента
  - История действий
- **Решение:**
  - В пользу клиента → возврат денег
  - В пользу партнёра → деньги партнёру
  - Комментарий с обоснованием

#### Финансы
- **Выплаты партнёрам:**
  - Автоматическое формирование выплат раз в неделю (понедельник)
  - Просмотр всех выплат (ожидают, обработаны, завершены)
  - Детализация по заказам
- **Статистика платформы:**
  - Общая выручка (gross)
  - Комиссия платформы
  - Количество заказов
  - Топ партнёров по выручке

#### Категории заведений
- Справочник категорий (пекарня, кафе, ресторан, магазин, отель)
- Добавление новых категорий
- Настройка иконок и цветов для карты

#### Аудит
- Лог действий администраторов:
  - Кто, когда, что сделал (одобрил партнёра, решил спор, заблокировал пользователя)
  - IP адрес

## 4. Бизнес-процессы

### 4.1 Жизненный цикл заказа

```
1. Клиент выбирает бокс
   ↓
2. Оплата через ЮKassa
   ↓ (статус: paid)
3. Партнёр должен подтвердить в течение 30 минут
   ├─ Подтвердил → статус: confirmed
   └─ НЕ подтвердил → автоотмена + возврат денег
   ↓
4. Клиент приходит в окно получения
   ↓
5. Сотрудник сканирует код → статус: picked_up
   ↓
6. Через 15 минут → автоподтверждение → статус: completed
   ├─ Клиент нажал "Не получил" в течение 15 мин → статус: disputed
   └─ Всё ок → завершён
   ↓
7. Клиент оставляет отзыв (опционально)
```

### 4.2 Статусы заказа

- **pending** — создан, ждёт оплаты (не используется, оплата сразу)
- **paid** — оплачен, ждёт подтверждения партнёра
- **confirmed** — партнёр подтвердил
- **picked_up** — сотрудник выдал бокс
- **completed** — успешно завершён
- **cancelled** — отменён (партнёр не подтвердил или клиент отменил)
- **refunded** — возврат денег выполнен
- **disputed** — спор открыт

### 4.3 Процесс выплат партнёрам

**Автоматическое создание выплат:**
- Каждый понедельник в 02:00 (cron job)
- За период: предыдущая неделя (понедельник-воскресенье)

**Алгоритм:**
1. Найти все completed заказы за неделю по каждому партнёру
2. Исключить заказы, которые уже в других выплатах
3. Рассчитать суммы:
   - Gross amount (общая выручка)
   - Commission rate (процент комиссии на момент выплаты)
   - Commission amount (сумма комиссии)
   - Net amount (к выплате партнёру)
4. Создать запись в `payouts`
5. Связать все заказы с выплатой через `payout_orders`
6. Отправить уведомление партнёру

**Статусы выплаты:**
- **pending** — сформирована, ожидает обработки
- **processing** — в процессе перевода
- **completed** — выплата завершена
- **failed** — ошибка при выплате

### 4.4 Процесс решения спора

1. Клиент нажимает "Не получил" в течение 15 минут после сканирования кода
2. Заказ → статус `disputed`
3. Создаётся запись в `disputes` с комментарием клиента
4. Администратор рассматривает спор:
   - Смотрит историю действий (кто, когда сканировал)
   - Может запросить доп. информацию у партнёра
5. Принимает решение:
   - **В пользу клиента:** возврат денег, партнёр без выручки
   - **В пользу партнёра:** деньги партнёру, клиент без возврата
6. Спор закрывается, уведомления обеим сторонам

## 5. Технические требования

### 5.1 Технологический стек

**Backend:**
- **Язык:** Go 1.24+
- **Архитектура:** Модульный монолит
- **База данных:** PostgreSQL 16+ (с PostGIS для геолокации)
- **Кеш:** Redis 7+
- **Брокер сообщений:** RabbitMQ 3.12+
- **API:** REST (возможно gRPC для внутренних сервисов)

**Frontend:**
- **Мобильное приложение:** Flutter 3.16+
- **Веб-панель партнёров:** React (или server-side HTML)
- **Админ-панель:** React

**Инфраструктура:**
- **Контейнеризация:** Docker, Docker Compose
- **Миграции:** golang-migrate
- **Логирование:** slog (стандартная библиотека Go)
- **Мониторинг:** Prometheus + Grafana (опционально)

**Внешние сервисы:**
- **Оплата:** ЮKassa API
- **SMS:** SMSC.ru / SMS.ru
- **Push-уведомления:** Firebase Cloud Messaging (FCM)
- **Карты:** Yandex Maps API
- **Хранилище файлов:** S3 / MinIO (для фото) / Yandex Cloud Object Storage

### 5.2 Нефункциональные требования

**Производительность:**
- Время ответа API: <200ms (95 percentile)
- Поддержка 1000 одновременных пользователей
- Обработка 10000+ заказов в день

**Доступность:**
- Uptime: 99.5% (допустимый downtime ~3.6 часа/месяц)
- Graceful degradation (если Redis упал, работаем без кеша)

**Безопасность:**
- HTTPS для всех соединений
- JWT токены с истечением (24 часа)
- Хеширование паролей (bcrypt, cost 12)
- Rate limiting (100 запросов/минуту на IP)
- Защита от CSRF, XSS, SQL Injection

**Масштабируемость:**
- Горизонтальное масштабирование API (stateless)
- Вертикальное масштабирование БД (до определённого предела)
- Очередь RabbitMQ для асинхронных задач

## 7. Монетизация

### Комиссия сервиса

**Базовая комиссия:** 20% от суммы заказа

**Промо-период:** первые 3 месяца — 10%

**VIP-тариф:** при >100 заказов/месяц — 15%

### Пример расчёта

```
Клиент купил бокс за 190₽
Комиссия 20% = 38₽
К выплате партнёру = 152₽
```

### Потенциальные доп. доходы (future)

- Платное продвижение заведений на карте
- Премиум-подписка для клиентов (без лимита заказов)
- Реклама партнёрских продуктов в приложении

## 8. Метрики успеха (KPI)

**Для клиентов:**
- MAU (Monthly Active Users) — месячная аудитория

- Средний чек
- Количество заказов на пользователя в месяц

**Для партнёров:**
- Количество активных партнёров
- Средняя выручка на партнёра
- Процент пропущенных подтверждений заказов (<10% — хорошо)

**Для платформы:**
- GMV (Gross Merchandise Volume) — общий объём транзакций
- Комиссионный доход
- Количество заказов в день
- Конверсия: регистрация → первый заказ

## 10. Риски и ограничения

### Технические риски

**Нагрузка на БД:**
- При росте до 100k+ пользователей может потребоваться шардинг
- Митигация: оптимизация запросов, индексы, read replicas

**Геолокация:**
- PostGIS может быть медленным при миллионах точек
- Митигация: кеширование в Redis, использование geo-хешей

**Оплаты:**
- Сбои ЮKassa могут блокировать заказы
- Митигация: retry механизм, альтернативный провайдер

### Бизнес-риски

**Недостаточно партнёров:**
- Без партнёров нет предложения для клиентов
- Митигация: активный B2B-маркетинг, промо-условия на старте

**Низкая конверсия клиентов:**
- Скачали приложение, но не делают заказы
- Митигация: gamification, push-уведомления, промокоды

**Проблемы с выполнением заказов:**
- Партнёры забывают собирать боксы, клиенты жалуются
- Митигация: автоотмена при неподтверждении, штрафы за пропуски

## 11. Словарь терминов

**Сюрприз-бокс (Surprise Box)** — набор продуктов, которые партнёр продаёт со скидкой. Содержимое заранее не известно точно.

**Партнёр** — владелец заведения, зарегистрированный на платформе.

**Заведение (Location)** — физическая точка партнёра (пекарня, кафе и т.д.).

**Фудшеринг** — практика передачи излишков еды вместо их выбрасывания.

**Окно получения** — временной интервал, в который клиент может забрать заказ (например, 18:00-19:00).

**Комиссия платформы** — процент от суммы заказа, который удерживает сервис.

**Gross amount** — общая выручка до вычета комиссии.

**Net amount** — сумма к выплате после вычета комиссии.

**GMV (Gross Merchandise Volume)** — общий объём транзакций через платформу.
